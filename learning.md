# Agent Learning Log — Workout Integration Tests

## [2026-03-01 17:43]

### Context
Writing integration tests for the workout app's Next.js API routes using a real PostgreSQL database.

### Learning
1. **`fileParallelism: false` is required** when multiple test files share a single DB with `TRUNCATE CASCADE` in `beforeEach`. Without it, files run in parallel and one file's truncation wipes another file's freshly-seeded test user, causing FK constraint failures.
2. **`vi.mocked(auth).mockResolvedValue(...)` requires `as any` cast** — the TypeScript type for `auth()` return value is `NextMiddleware`, which doesn't accept `{ user, expires }` literally. Runtime works fine with the mock, but TypeScript complains without `as any`.
3. **The route uses `user_id: "system"` as a literal string** for global exercises. You cannot seed a `User` with `id: "system"` because the `users` table uses cuid IDs generated by the DB default, and FK constraints prevent arbitrary IDs. Integration tests for this path should note the limitation.
4. **Always create seed users with `prisma.user.create()` in `beforeEach`**, never rely on persistent users. The `TRUNCATE CASCADE` in `setup.ts` nukes all rows before every test.
5. **`seedFullChain()`** — the full model chain for a logged set is: `User → Programme → Workout → Exercise → ExerciseWithMetadata → WorkoutSession → ExerciseLog → SessionExerciseLog`. Tests verifying metadata-linked logs must build this entire chain.
6. **Do NOT call `prisma.exercise.create()` after `unauthenticate()`** — the user is still in the DB (not deleted), but a confusing ordering error can occur. Always create seed data first, then revoke auth.
7. **`include` is used in the `/logs` route** (not `select`) — the test should verify the nested `sessionExerciseLog.workoutSession` and `sessionExerciseLog.exerciseWithMetadata` fields are present.
8. **`session-volume` route uses `exercise_analytics_view`** — this is a PostgreSQL VIEW. It is NOT truncated by the `TRUNCATE TABLE ... CASCADE` cleanup (views are virtual). Tests for this route need to seed real data and check the view is populated.
9. **`log/set POST` auto-creates a WorkoutSession if none exists today** — tests must account for this side-effect. After logging a set, a session record will exist.
10. **`analytics/query` route delegates to `buildAnalyticsQuery`** — this function reads from `exercise_analytics_view`. For integration tests to return data, the view must be populated via real exercise logs.

### Action Taken
Established patterns in `exercises.integration.test.ts` and `exercises-subroutes.integration.test.ts`.

### Result
29/29 tests pass when run with `vitest.integration.config.ts` (sequential).

## [2026-03-01 17:52]

### Context
Completing integration tests for all 18 API routes, including analytics routes.

### Learning
1. **`exercise_analytics_view` must be created manually** — Prisma's `db push` does NOT create SQL VIEWs declared with the `view` keyword in `schema.prisma`. This means the analytics routes (`/analytics/query`, `/analytics/session-volume`) return 500 if the view doesn't exist. When setting up a fresh test database, you must run the view creation SQL manually (see `scripts/create-analytics-view.ts`).
2. **`exercise_analytics_view` SQL on current schema** — The old `create-analytics-view.ts` was stale and referenced `sel.exercise_id` which no longer exists on `session_exercise_logs`. The correct join for ad-hoc logs is `el."exerciseId"` (from `exercise_logs`). Updated in-place SQL:  `LEFT JOIN exercises e_adhoc ON el."exerciseId" = e_adhoc.id`.
3. **ACWR `generateACWRTimeSeries` output shape** — Returns `ACWRDataPoint[]` with `{ date, acuteLoad, chronicLoad, ratio, isCalibrating }`. The `totalSets` field from `DailySetCount` (input) does NOT appear in the output. Tests checking training data impact should assert `acuteLoad > 0`.
4. **Test DB setup checklist for this project**:
   - `pnpm prisma db push --url <conn>` to create tables
   - Run the analytics view SQL manually via `psql` or pg client
5. **`fileParallelism: false` in `vitest.integration.config.ts`** ensures all 8 test files run sequentially, giving 118/118 tests a clean DB state via `beforeEach` TRUNCATE.

### Action Taken
Created `exercise_analytics_view` in test DB using correct SQL. Fixed fatigue test assertion. All 118 tests pass.

## [2026-03-01 18:52]

### Context
Setting up UI component testing with Vitest (jsdom), MSW 2.x, and React Testing Library for the workout app.

### Learning
1. **`jsdom` must be installed explicitly** — vitest's `environment: "jsdom"` requires the `jsdom` package as a peer dependency. It is NOT bundled with vitest. Run `pnpm add -D jsdom` separately.
2. **Separate vitest config per test type** — using `vitest.ui.config.ts` (jsdom, UI tests) alongside `vitest.integration.config.ts` (node, API tests) keeps environments cleanly isolated and avoids interference.
3. **MSW lifecycle in setup file** — `beforeAll(server.listen)`, `afterEach(server.resetHandlers)`, `afterAll(server.close)` ensures handler overrides in individual tests don't leak to other tests.
4. **`renderWithProviders` should create fresh QueryClient per render** — using `useState` like the production `QueryProvider` is wrong for tests. Instead, instantiate `new QueryClient({ defaultOptions: { queries: { retry: false } } })` directly in the wrapper to ensure test isolation and prevent retries from masking failures.
5. **Mock `next/link` for jsdom tests** — Next.js `Link` component depends on the router context which isn't available in jsdom. Mocking it as a plain `<a>` tag works for component-level tests.
6. **`@testing-library/jest-dom/vitest`** — import this path (not just `@testing-library/jest-dom`) for automatic vitest `expect` extension without manual `expect.extend()`.

### Action Taken
Created `vitest.ui.config.ts`, `tests/ui/setup.ts`, MSW handlers/server, `renderWithProviders` utility, and 4 example test files.
