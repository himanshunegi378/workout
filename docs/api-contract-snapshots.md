# API Contract Snapshots

These snapshots capture behavior from the current Next.js route source. Route source is authoritative when it differs from `docs/api-documentation.md`.

## Snapshot Status Legend
- `Captured`: source behavior has been documented.
- `Implemented`: equivalent NestJS code exists.
- `Verified`: NestJS behavior has been compared with Next.js behavior.
- `Retained in Next.js`: endpoint is intentionally kept on Next.js because migrating it would change framework/session ownership.
- `Needs confirmation`: behavior or migration ownership is unclear.

## Auth

### POST `/api/auth/signup`

Status: `Verified`

Current source file: `app/api/auth/signup/route.ts`

Target NestJS controller/service: `auth/AuthController.signup` + `AuthService.signup`

Auth requirement: Public. No session or user ID is required.

Request params: None.

Query params: None.

Request body:
- `username`: required string.
- `password`: required string.
- Extra body fields are ignored.

Success status code: `201`

Success response shape:

```ts
{
  id: string;
  username: string;
}
```

Error status codes and response shapes:
- `400`: `{ "error": "Username must be at least 3 characters" }`
- `400`: `{ "error": "Password must be at least 6 characters" }`
- `409`: `{ "error": "Username already taken" }`
- `500`: `{ "error": "Something went wrong. Please try again." }`

Validation behavior:
- Parses request JSON with `await request.json()`.
- Username is invalid if missing, not a string, or `username.trim().length < 3`.
- Password is invalid if missing, not a string, or `password.length < 6`.
- Password validation does not trim before measuring length.
- Username is normalized with `username.trim().toLowerCase()` for duplicate lookup and creation.
- Duplicate checks are case-insensitive because both stored and incoming signup usernames are normalized to lowercase.

Ownership/user-scoping behavior:
- Public account creation only. No user-scoped reads are involved.

Idempotency behavior:
- No client-generated ID idempotency.
- Repeating a successful signup with the same normalized username returns `409`.

Database reads/writes:
- Reads `prisma.user.findUnique({ where: { username: normalizedUsername } })`.
- Writes `prisma.user.create({ data: { username: normalizedUsername, password_hash } })`.
- `User.id` is generated by Prisma/database defaults.
- `User.created_at` is generated by Prisma/database defaults.

Side effects:
- Hashes the raw password using `bcrypt.hash(password, 12)`.
- Logs caught errors with `console.error("Signup error:", error)`.

Date/time behavior:
- No explicit date input or date response.
- `created_at` is implicitly generated but not returned.

Special compatibility notes:
- Response must not include `password_hash`.
- Existing tests assert username lowercasing, duplicate case-insensitivity, persisted hashed password, and `201`/`400`/`409` statuses.
- NestJS implementation should avoid DTO transforms that trim or lowercase password, reject unknown fields, or alter the exact validation messages.

Frontend caller references:

- `app/features/exercises/api/mutation-hooks/use-create-exercise.ts`

Success status codes:
- `201`: created a new custom exercise.
- `200`: idempotent replay with an existing exercise owned by the authenticated user.

Success response shape:

```ts
{
  id: string;
  name: string;
  description: string | null;
  muscle_group: "Abs" | "Back" | "Biceps" | "Cardio" | "Chest" | "Forearms" | "Legs" | "Shoulders" | "Triceps";
  is_global: boolean;
  user_id: string | null;
}
```

Create success example:

```json
{
  "id": "custom_exercise_id",
  "name": "Bench Press",
  "description": null,
  "muscle_group": "Chest",
  "is_global": false,
  "user_id": "user_123"
}
```

Error status codes and response shapes:
- `400`: `{ "error": "Exercise name is required" }`
- `400`: `{ "error": "Valid muscle group is required" }`
- `401`: `{ "error": "Unauthorized" }`
- `403`: `{ "error": "Forbidden" }`
- `500`: `{ "error": "Failed to create exercise" }`

Validation behavior:
- Parses request JSON with `await request.json()`.
- Auth check happens before JSON parsing.
- `name` is invalid if missing, not a string, or `name.trim().length === 0`.
- Idempotency check happens after name validation and before muscle group validation.
- `muscle_group` is invalid if missing or not included in `Object.values(MuscleGroup)`.
- `description` is not type-checked; persistence uses `description || null`.

Ownership/user-scoping behavior:
- Created exercise always gets `user_id` set to the authenticated user ID and `is_global: false`.
- If a submitted `id` already exists and the record is global or owned by another user, returns `403`.
- If a submitted `id` already exists and belongs to the user, returns the existing full exercise record with status `200`.

Idempotency behavior:
- Truthy `id` triggers `prisma.exercise.findUnique({ where: { id } })`.
- Existing owned record returns `200` and does not validate `muscle_group`.
- Existing global or foreign-owned record returns `403`.
- Falsy `id` is ignored and Prisma generates the ID.

Database reads/writes:
- Reads `prisma.exercise.findUnique` only when `id` is truthy.
- Writes `prisma.exercise.create` with `id: id || undefined`, trimmed `name`, `description: description || null`, `muscle_group`, `user_id`, and `is_global: false`.

Side effects:
- Creates one row in `exercises` for non-idempotent success.

Date/time behavior: None.

Special compatibility notes:
- `name` is trimmed for create responses because the persisted record is returned.
- `description: ""`, `0`, or `false` becomes `null` because the source uses `description || null`.
- Existing tests assert create success, persistence, description persistence, name validation, muscle group validation, unauthenticated `401`, scoping, ordering, and selected GET keys.

- Missing normalized exercise IDs returns `400`.
- `from` and `to` are parsed with `new Date(value)`; invalid dates return route-specific `400` messages.
- If both dates exist and `from > to`, returns `` `from` must be before or equal to `to` ``.

Ownership/user-scoping behavior:
- Logs are filtered by `user_id = session.user.id`.
- The route does not separately verify exercise ownership/global visibility.

Database reads/writes:
- Reads `prisma.exerciseLog.findMany`.
- Filters direct `exerciseId in exerciseIds` or linked metadata `exercise_id in exerciseIds`.
- Applies date filters to workout session date for linked logs, and `ExerciseLog.date` for ad-hoc logs with `sessionExerciseLog is null`.
- Orders by workout session date descending, then log date ascending.
- Selects log fields plus nested workout session date/start time and metadata prescription fields.
- No writes.

Side effects: None.

Date/time behavior:
- Public `workoutSession.date` and `workoutSession.start_time` are ISO strings.
- Ad-hoc logs with no session use `log.date.toISOString()` and `start_time: null`.

Special compatibility notes:
- Response hides Prisma's `sessionExerciseLog` join model.
- `exerciseId` in the response falls back to linked metadata `exercise_id` when direct `ExerciseLog.exerciseId` is null.

- `401`: `{ "error": "Unauthorized" }`

Validation behavior:

Ownership/user-scoping behavior:

- Can update other programmes and activity logs when creating an active programme.

Date/time behavior:
- Uses `new Date()` for closing old activity logs and starting a new activity log.

Special compatibility notes:
- `is_active` side effects are based on truthiness, not strict boolean validation.
- Owned idempotent replay returns before active-programme side effects.
